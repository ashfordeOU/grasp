Grasp Documentation

Turn any GitHub repository or local codebase into an interactive architecture map in seconds — dependency graphs, health scores, security scanning, and 48 MCP tools for AI agents. No setup, no accounts, no data leaving your machine.

Browser App

The browser app runs entirely client-side — no server, no proxy, no build step. Your code never leaves your machine.

Analyzing public repositories

Paste a GitHub owner/repo slug or a full URL into the input field and press Enter or click Analyze:

facebook/react
https://github.com/vercel/next.js

Public repos use GitHub's unauthenticated API (60 requests/hour). For large codebases, add a token for 5,000 requests/hour.

Private repositories

  1. Go to github.com/settings/tokens and create a Personal Access Token with repo scope.
  2. In the browser app, open the auth dropdown and select Token.
  3. Paste your token. It is stored in localStorage only — never sent anywhere except directly to the GitHub API.
  4. Analyze any private repo you have access to.

Local files

Two options to analyze files on your machine:

  • Drag & drop — drag a folder directly onto the browser app window.
  • 📁 Open Folder button — click it to open a native folder picker (uses the File System Access API). Works offline, no internet required.

Shareable links

After any analysis, click the 🔗 button in the top bar to copy a link. Anyone with the link can re-run the same analysis. For embeds (iframe, README badge, React snippet), use ⋯ → 🔗 Embed.

Offline support: Local file analysis works without an internet connection. GitHub repo analysis requires internet access to fetch code via the GitHub API.

CLI Reference

Install the CLI globally or run one-off with npx. Outputs a colour-coded health report and writes grasp-report.json. Exit code 0 = CI pass, 1 = issues found.

# Install once
npm install -g grasp-mcp-server

# Run without installing
npx --package=grasp-mcp-server grasp .

MCP Server Setup

Grasp ships a Model Context Protocol (MCP) server that exposes the full analysis engine as callable tools for Claude Code, Cursor, and any MCP-compatible AI agent. 48 tools

Installation

npm install -g grasp-mcp-server

Config for Claude Code, Cursor, and other MCP clients

Add this to your MCP settings file (e.g. ~/.claude/claude_mcp_settings.json, .cursor/mcp.json, etc.):

{
  "mcpServers": {
    "grasp": {
      "command": "npx",
      "args": ["-y", "grasp-mcp-server"]
    }
  }
}
Tip: Using npx -y means the MCP client always runs the latest published version without a separate install step.

GitHub Actions

Add Grasp to your CI pipeline to get automatic health reports on every pull request.

Automatic PR health comments

This reusable workflow posts (and updates) a health comment on every opened or updated PR:

# .github/workflows/grasp.yml
name: Grasp Health Check
on:
  pull_request:
    types: [opened, synchronize, reopened]
jobs:
  health:
    uses: ashfordeOU/grasp/.github/workflows/grasp-health.yml@main

The workflow posts a comment that updates on every re-push, showing:

  • Health score with a progress bar (e.g. ████████░░ 82/100)
  • Grade (A–F), file count, function count
  • Architecture issues, circular deps, security findings
  • Blast radius for the changed files in this PR

Architecture rules in CI

Use the --check flag to enforce grasp.yml rules and fail the build on violations:

- name: Enforce architecture rules
  run: npx grasp-mcp-server . --check

CLI-based CI gate (JSON)

- name: Check Grasp health
  run: |
    PASSED=$(cat grasp-report.json | jq '.ci.passed')
    SCORE=$(cat grasp-report.json | jq '.ci.score')
    echo "Health score: $SCORE"
    if [ "$PASSED" != "true" ]; then
      echo "Grasp CI check failed"
      cat grasp-report.json | jq '.ci.failures'
      exit 1
    fi

Architecture Rules (grasp.yml)

Define architecture standards in a grasp.yml file at your repo root. Grasp enforces them in CI and surfaces violations as issues in the browser app.

Example config

rules:
  - min_health_score: 70    # fail if health score drops below 70
  - max_blast_radius: 20    # flag any file that affects 20+ others

How it works

  1. Create grasp.yml in your project root with the rules above.
  2. Run grasp . --check locally to see violations.
  3. Add the GitHub Actions workflow to enforce rules on every PR.
  4. Exit code 1 is returned when any rule is violated — blocking the merge.
ci.passed is set to false when: health score < threshold, any circular dependencies exist, or any high-severity security issues are found.

Violations in JSON export

Architecture rule violations appear in the issues array of the JSON export:

{
  "type": "critical",
  "title": "N Architecture Violations",
  "items": [
    {
      "name": "utils → services",
      "file": "src/utils/api.js",
      "toFile": "src/services/auth.js",
      "fn": "authenticate"
    }
  ]
}

Visualization Modes

Grasp provides 9 graph layout types and 13 color modes. Mix and match to answer different architectural questions.

Graph Types

  • Graph Force-directed dependency graph — drag, zoom, click to explore
  • 3D Graph Three-dimensional force graph — rotate, pan, zoom through your architecture
  • Arch Layer-by-layer architecture diagram with zoom/pan
  • Treemap Files sized by line count, grouped by folder
  • Matrix Adjacency matrix showing all file-to-file dependencies
  • Tree Hierarchical cluster dendrogram
  • Flow Folder-level dependency flow (Sankey diagram)
  • Bundle Circular layout with arc-based connections
  • Cluster Separated force graphs per folder

Color Modes

  • Folder Color by directory structure
  • Layer Color by architectural layer (UI, Services, Utils, etc.)
  • Churn Color by commit frequency — red = most-changed hot spots
  • Depth Color by max brace-nesting depth
  • Complexity Color by cyclomatic complexity (green → yellow → red)
  • Blast Color by blast radius impact for a selected file
  • Dup Color by duplicate code density
  • Owner Color by top contributor — spot bus-factor risks
  • Issues Color by number of linked GitHub Issues per file
  • Coverage Color by test coverage — highlight untested files
  • Bundle Color by bundle size contribution
  • API Surface Color by API endpoint exposure
  • Runtime Color by actual runtime call frequency from a live trace

Privacy & Security

Your code stays on your machine. Grasp is designed from the ground up with a privacy-first architecture.

🌐
Browser app — 100% client-side Runs entirely in your browser. No server, no proxy. GitHub API calls go directly from your browser to GitHub — nothing passes through Ashforde servers.
🔑
GitHub token in localStorage only Your Personal Access Token never leaves your machine. It is stored in localStorage and sent only to the GitHub API directly from your browser.
📡
No analytics or tracking The browser app contains no analytics, no telemetry, no tracking pixels, and no account system. There is nothing to sign up for.
🤖
MCP server — local subprocess only Runs as a local subprocess. The only outbound connections it makes are to the GitHub API when you explicitly analyze a GitHub repo. Local directory analysis stays entirely in memory.
📂
Local files never leave your machine When analyzing local files, all parsing and processing happens in your browser or in the local CLI process. No file content is ever uploaded.
🔍
Open source — audit it yourself The entire browser app is one open-source HTML file. Read every line of code that runs in your browser at github.com/ashfordeOU/grasp.
